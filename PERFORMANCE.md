# PERFORMANCE

Этот документ фиксирует backlog по производительности и масштабируемости DormEX.

Цель:

- снизить число запросов к БД и API на одного активного пользователя
- убрать архитектурные решения, которые плохо масштабируются по числу соединений
- сократить latency на типовых сценариях
- подготовить проект к более высокой конкурентной нагрузке

Принципы приоритизации:

- `P0` - изменения, которые уже сейчас дают наибольший выигрыш и убирают главные bottleneck'и
- `P1` - изменения второго эшелона, которые нужны после P0
- `P2` - улучшения, которые полезны после стабилизации базовой архитектуры

  Зачем: без baseline невозможно доказать, что оптимизация реально помогла.
  Критерий готовности: есть повторяемый сценарий замера и сохранённые результаты до оптимизаций.

- [ ] Добавить SQL query logging и счётчик запросов на запрос API в dev/staging.
  Что сделать: логировать duration SQL-запросов, endpoint, суммарное количество SQL-команд на HTTP request.
  Зачем: быстро находить N+1 и избыточные чтения.
  Критерий готовности: для каждого ключевого endpoint видно число SQL-запросов и самые медленные запросы.

- [ ] Добавить нагрузочные сценарии для ключевых пользовательских потоков.
  Что сделать: подготовить сценарии для ленты задач, страницы задачи, чатов, уведомлений, откликов и аналитики.
  Зачем: проверять изменения не на ощущениях, а на одинаковой нагрузке.
  Критерий готовности: есть как минимум один автоматизированный сценарий с 20, 100 и 500 виртуальными пользователями.

- [ ] Ввести отдельный performance checklist для PR.
  Что сделать: добавить в процесс разработки вопросы про новые polling-механизмы, новые count-запросы, новые full-table scans и новые refetch cascade на frontend.
  Зачем: не допускать регрессий после исправлений.
  Критерий готовности: checklist используется в review и покрывает backend, frontend и БД.

## P0. Соединения с БД и модель доступа к Postgres

- [ ] Заменить `psycopg.connect(...)` на пул соединений.
  Что сделать: использовать `psycopg_pool.ConnectionPool` или async-пул, и выдавать соединения из пула вместо нового подключения на каждую операцию.
  Зачем: текущее открытие нового соединения на каждый запрос создаёт лишнюю нагрузку на Postgres и повышает latency.
  Критерий готовности: в коде больше нет паттерна создания нового физического подключения на каждый repository call.

- [ ] Настроить размер пула соединений и таймауты.
  Что сделать: вынести в env параметры `min_size`, `max_size`, acquisition timeout, statement timeout, idle timeout.
  Зачем: чтобы приложение не упиралось либо в starvation пула, либо в слишком большой fan-out к базе.
  Критерий готовности: настройки документированы и используются в runtime.

- [ ] Пройтись по репозиториям и убрать лишние открытия connection внутри одной бизнес-операции.
  Что сделать: если одна операция логически атомарна, выполнять её в одном connection/transaction scope.
  Зачем: сейчас часть методов внутри одной операции повторно заходит в БД через новые connection boundaries.
  Критерий готовности: критические write-flow работают в одном transaction scope.

- [ ] Установить `statement_timeout` для проблемных запросов.
  Что сделать: ограничить время выполнения тяжёлых SQL, чтобы один плохой запрос не блокировал систему.
  Зачем: under load зависающие запросы быстро ухудшают весь сервис.
  Критерий готовности: длительные запросы прерываются контролируемо и попадают в лог.

- [ ] Проверить совместимость с PgBouncer.
  Что сделать: подготовить backend к работе через внешний pooler на staging/prod.
  Зачем: при росте нагрузки внешний pooler часто нужен даже при локальном пуле драйвера.
  Критерий готовности: задокументирован режим работы с PgBouncer и ограничения транзакций.

## P0. Полный пересмотр realtime

- [ ] Убрать polling-based realtime на backend.
  Что сделать: заменить модель "каждый websocket раз в 2 секунды опрашивает БД" на push-модель по событию.
  Зачем: текущий подход масштабируется пропорционально числу websocket-подключений и постоянно нагружает БД даже без изменений данных.
  Критерий готовности: websocket не делает периодические snapshot-запросы в базу по таймеру.

- [ ] Выбрать транспорт событий для realtime.
  Что сделать: принять решение между PostgreSQL `LISTEN/NOTIFY`, Redis Pub/Sub, Redis Streams или отдельным event bus.
  Зачем: нужна нормальная доставка событий между write-path и websocket hub.
  Критерий готовности: выбран и документирован единый механизм публикации событий.

- [ ] Сделать единый realtime hub / event dispatcher.
  Что сделать: отделить доставку событий клиентам от слоя контроллеров и репозиториев.
  Зачем: сейчас realtime логика смешана с API-слоем и плохо масштабируется.
  Критерий готовности: события публикуются в один слой, а websocket endpoint только подписывает клиента.

- [ ] Передавать по websocket дельты, а не полные snapshots.
  Что сделать: отправлять событие вида `task.updated`, `offer.created`, `notification.read`, `chat.message.created`, а не полный список сущностей.
  Зачем: это резко сокращает SQL, сериализацию и сетевой трафик.
  Критерий готовности: payload события содержит только изменившиеся данные или идентификаторы для точечного обновления cache.

- [ ] Разделить каналы realtime по предметной области и гранулярности.
  Что сделать: отдельные каналы для ленты задач, конкретной задачи, конкретного чата, счётчика уведомлений и списка уведомлений.
  Зачем: сейчас один канал зачастую тянет больше данных, чем нужно конкретному экрану.
  Критерий готовности: подписка клиента включает только реально нужный ему поток событий.

- [ ] Убрать пересчёт SHA/signature на полном snapshot для каждого websocket.
  Что сделать: перестать сериализовать и хешировать весь payload на каждом тике.
  Зачем: это лишний CPU и аллокации памяти, особенно на больших массивах сообщений и задач.
  Критерий готовности: сигнатуры полных snapshots больше не используются как основной механизм детекции изменений.

- [ ] Вынести realtime-права доступа в отдельный лёгкий guard.
  Что сделать: проверять доступ к каналу при подписке и не повторять тяжёлые проверки в каждом цикле.
  Зачем: уменьшить число повторных чтений контекста и сущностей.
  Критерий готовности: подписка проверяется один раз, дальнейшая доставка событий не требует повторных full reads.

## P0. Удаление двойного polling и каскадных refetch на frontend

- [ ] Убрать `refetchInterval` в экранах, где уже используется websocket.
  Что сделать: пройтись по `Index`, `TaskDetail`, `OfferNegotiation`, `TopNav` и убрать дублирующий polling.
  Зачем: сейчас frontend одновременно использует HTTP polling и websocket, что кратно раздувает нагрузку.
  Критерий готовности: для одной и той же сущности не существует двух параллельных механизмов постоянного обновления.

- [ ] Убрать массовые `refetchQueries(...)` после websocket-событий.
  Что сделать: заменить широкие refetch cascade на `setQueryData`, точечную invalidation и event-specific updates.
  Зачем: один realtime event сейчас может запускать десятки лишних запросов.
  Критерий готовности: событие уведомления не инициирует refetch несвязанных сущностей.

- [ ] Ввести карту инвалидации по типам событий.
  Что сделать: для каждого события описать, какие query keys реально затрагиваются.
  Зачем: это предотвращает случайные refetch "на всякий случай".
  Критерий готовности: есть единая таблица соответствия `event -> affected queries`.

- [ ] Настроить более длинные `staleTime` и `gcTime` для редко меняющихся данных.
  Что сделать: увеличить время жизни cache для справочников, профиля, списка общежитий и части аналитики.
  Зачем: сейчас часть данных можно не перезапрашивать так часто.
  Критерий готовности: редко меняющиеся query не уходят повторно в сеть без необходимости.

- [ ] Оставить polling только как controlled fallback.
  Что сделать: если websocket недоступен, использовать редкий fallback polling с большим интервалом и backoff.
  Зачем: fallback нужен, но не как режим по умолчанию.
  Критерий готовности: при живом websocket основной polling выключен.

- [ ] Ввести debounce/throttle для пользовательского поиска и фильтров.
  Что сделать: уменьшить число запросов при изменении строки поиска и фильтров.
  Зачем: это снижает burst-нагрузку на `/tasks`.
  Критерий готовности: поиск не отправляет запрос на каждый символ без debounce.

## P0. Оптимизация самых горячих backend endpoint'ов

- [ ] Пересобрать `/ws/tasks`, чтобы он не читал полную ленту задач.
  Что сделать: если экрану нужен только факт изменения или count, отдавать только это.
  Зачем: сейчас ради `total` читается гораздо больше данных, чем реально используется.
  Критерий готовности: endpoint не вызывает тяжёлый `list_tasks(limit=100)` для heartbeat-like сценария.

- [ ] Пересобрать `/ws/tasks/{task_id}`.
  Что сделать: разделить `task.updated` и `offers.updated`, сократить число чтений task detail и списка offers.
  Зачем: сейчас канал на одну задачу делает дорогие aggregate reads даже когда изменился только один offer.
  Критерий готовности: изменение offer не приводит к полному перечитыванию всего task detail без необходимости.

- [ ] Пересобрать `/ws/chats/{chat_id}`.
  Что сделать: перейти на доставку новых сообщений по `message_id` или `created_at`, а историю читать отдельно пагинированным endpoint'ом.
  Зачем: сейчас websocket перечитывает до 100 сообщений повторно.
  Критерий готовности: realtime чата передаёт только новые сообщения и события read-state.

- [ ] Разделить счётчик уведомлений и список уведомлений.
  Что сделать: для колокольчика держать дешёвый канал/endpoint с `unread_count`, а тяжёлый список грузить только при открытии.
  Зачем: count и список имеют разную стоимость и разный UX-смысл.
  Критерий готовности: закрытый popover уведомлений не тянет полный список без повода.

- [ ] Пересмотреть `list_tasks`.
  Что сделать: уменьшить стоимость `COUNT(*)`, оценить cursor pagination вместо `OFFSET`, проверить нужность total для всех сценариев.
  Зачем: `COUNT(*) + items query` на каждой загрузке страницы ленты плохо масштабируется.
  Критерий готовности: для больших выборок latency ленты меньше, а число прочитанных строк не растёт линейно с глубиной страниц.

- [ ] Пересмотреть `get_task_detail`.
  Что сделать: убрать повторные запросы, объединить часть данных в один запрос или в ограниченный батч.
  Зачем: task detail сейчас собирается из нескольких последовательных SQL-команд.
  Критерий готовности: число SQL-запросов на detail page уменьшено и зафиксировано метриками.

  Критерий готовности: список задач пользователя выполняет фиксированное число запросов независимо от числа completed items.

- [ ] Пересмотреть `_build_task_review_summary`.
  Что сделать: минимизировать повторное чтение assignment и reviews, вынести batch-версию.
  Зачем: один и тот же summary используется в нескольких местах и дорог в пересчёте.

- [ ] Сократить повторные проверки доступа к chat/task/offer.
  Что сделать: там, где сущность уже прочитана в текущем request scope, не читать её заново через новый repository call.
  Зачем: сейчас часть методов делает дополнительные read-before-write проверки отдельными запросами.

- [ ] Пересмотреть pattern "write -> commit -> re-read full entity".
  Что сделать: где возможно, использовать `RETURNING` и собирать ответ без дополнительного полного чтения.
  Зачем: это уменьшает число round-trip к БД.
  Критерий готовности: create/update операции на горячих сущностях требуют меньше SQL-команд.

- [ ] Сократить лишние `COUNT(*)` в write-flow.
  Что сделать: проверить, где можно поддерживать counters инкрементально, а не пересчитывать каждый раз.
  Зачем: repeated full counts под нагрузкой быстро становятся дорогими.
  Критерий готовности: самые частые write-path больше не делают полные пересчёты без острой необходимости.

- [ ] Ввести правило: каждый новый сложный запрос сопровождается `EXPLAIN ANALYZE`.

## P1. Транзакции, консистентность и write-path

- [ ] Нормализовать transaction boundaries в write-flow.
  Что сделать: явным образом определить, какие операции должны происходить в одной транзакции, а какие можно вынести.

  Что сделать: вместо полного пересчёта метрик после ряда событий перейти на инкрементальные обновления либо фоновые reconcile job.
  Зачем: `_refresh_user_metrics` сейчас делает несколько массовых update/select операций.
  Критерий готовности: частые write-flow не запускают тяжёлый полный пересчёт пользовательских агрегатов.

- [ ] Пересмотреть обновление `offers_count` и task status.
  Что сделать: оценить хранение счётчиков через инкрементальные операции, триггеры или специализированный service layer.
  Зачем: repeated recount по `task_offers` в горячих путях будет дорог при большом количестве откликов.

- [ ] Разделить быстрые write-path и тяжёлые побочные эффекты.

  Зачем: пользовательский write request не должен ждать всё вторичное обслуживание.
  Критерий готовности: p95 на create/send/accept снижен за счёт выноса побочных действий.

- [ ] Ввести идемпотентность для чувствительных операций.
  Что сделать: защитить повторные submit на frontend/backend там, где возможны дубликаты из-за ретраев.
  Зачем: под нагрузкой и при нестабильной сети дубликаты усиливают и бизнес-риск, и нагрузку.
  Критерий готовности: повтор одного и того же запроса не создаёт каскад лишних записей.

  Критерий готовности: открытие длинного чата не тянет слишком много сообщений сразу.

- [ ] Разделить "новое сообщение" и "mark as read".
  Что сделать: не вызывать `markRead` после каждого realtime message без дополнительных условий.
  Зачем: сейчас read-state может избыточно дёргать backend.
  Критерий готовности: read acknowledgements происходят реже и осмысленно.

- [ ] Ввести unread counters для чатов.
  Что сделать: считать непрочитанные сообщения инкрементально и отдавать summaries, а не полные сообщения для определения состояния.
  Зачем: это уменьшает количество чтений истории чатов.
  Критерий готовности: список чатов отображает unread state без чтения последних 50-100 сообщений.

- [ ] Оптимизировать список уведомлений.
  Что сделать: поддерживать lightweight summary для dropdown и отдельную детальную выдачу для полной страницы/экрана.
  Зачем: popup не должен требовать тяжёлого payload.
  Критерий готовности: размер payload и latency dropdown уведомлений снижены.

- [ ] Добавить ограничение размера payload для websocket event'ов.
  Что сделать: ввести верхние лимиты на размер списков и body event'ов.
  Зачем: один большой event не должен забивать канал и клиентский main thread.
  Критерий готовности: websocket payload bounded и валидируется.

## P1. Аналитика

- [ ] Пересмотреть runtime-расчёт аналитики категорий.
  Что сделать: оценить materialized views, precomputed tables или периодический refresh агрегатов.
  Зачем: аналитика хорошо подходит под предрасчёт, особенно если используется на нескольких экранах.
  Критерий готовности: категория аналитики не строится целиком из сырых данных на каждый клиентский запрос.

- [ ] Кэшировать ответы аналитики.
  Что сделать: добавить application cache с TTL и инвалидировать его по событиям завершения сделок.
  Зачем: аналитика меняется редко относительно частоты чтения.
  Критерий готовности: повторные запросы аналитики обслуживаются из cache.

- [ ] Снизить fan-out запросов на странице аналитики.
  Что сделать: либо объединить нужные категории в один backend endpoint, либо делать batched fetch.
  Зачем: режим "all categories" сейчас легко превращается в несколько одновременных запросов.
  Критерий готовности: число API-запросов для экрана аналитики уменьшено.

## P2. Архитектурная декомпозиция и качество кода

- [ ] Разбить `platform_repository.py` на доменные репозитории.
  Что сделать: разделить задачи, отклики, переговоры, чаты, уведомления, аналитика, профиль, auth.
  Зачем: текущий монолитный repository затрудняет локальную оптимизацию и повышает риск случайных регрессий.
  Критерий готовности: hot-path код находится в небольших доменных модулях с понятной ответственностью.

- [ ] Отделить query-model и command-model.
  Что сделать: рассмотреть CQRS-lite для тяжёлых read-сценариев.
  Зачем: разные требования к чтению и записи сейчас смешаны в одном слое.
  Критерий готовности: дорогие read-model можно развивать независимо от write транзакций.

- [ ] Ввести слой domain events.
  Что сделать: описать события `offer_created`, `offer_accepted`, `chat_message_sent`, `review_created`, `notification_created`.
  Зачем: это упростит push realtime, фоновые задачи и аналитические invalidate.
  Критерий готовности: write-path публикует события, а не дёргает всё синхронно.

- [ ] Добавить фоновый worker для тяжёлых вторичных задач.
  Что сделать: вынести часть побочных эффектов в background job runner.
  Зачем: синхронный request-response не должен нести всю вторичную нагрузку.
  Критерий готовности: тяжёлые побочные операции можно выполнять вне пользовательского HTTP запроса.

- [ ] Выделить performance-sensitive модули и покрыть их тестами регрессии.
  Что сделать: тесты на число SQL-запросов, на отсутствие лишнего polling, на корректную invalidation cache.
  Зачем: иначе часть проблем быстро вернётся.
  Критерий готовности: есть автотесты не только на бизнес-логику, но и на performance invariants.

## P2. Инфраструктура и эксплуатация

- [ ] Добавить application metrics.
  Что сделать: собирать Prometheus-метрики по latency, active websocket connections, DB pool usage, SQL duration, cache hit ratio.
  Зачем: без операционных метрик high-load проблемы невозможно нормально сопровождать.
  Критерий готовности: есть дешёвый dashboard по backend и БД.

- [ ] Добавить tracing.
  Что сделать: трассировать путь запроса через controller -> service -> repository -> SQL.
  Зачем: это помогает увидеть, где именно тратится время.
  Критерий готовности: trace показывает структуру горячих запросов и число SQL внутри них.

- [ ] Настроить алерты на деградацию производительности.
  Что сделать: алерты на p95 latency, saturation DB pool, число открытых websocket, error rate и slow SQL.
  Зачем: проблемы надо ловить до пользовательских жалоб.
  Критерий готовности: алерты приходят при контролируемом ухудшении стенда.

- [ ] Провести отдельный capacity planning.
  Что сделать: оценить, сколько пользователей выдерживает текущая конфигурация backend, Postgres и nginx после P0/P1.
  Зачем: оптимизация должна завершаться не только кодом, но и понятной оценкой пределов.
  Критерий готовности: есть таблица "нагрузка -> ресурсы -> latency -> bottleneck".

## Предлагаемый порядок выполнения

1. Измеримость: baseline, query logging, нагрузочные сценарии.

- [ ] Backend использует пул соединений.
- [ ] Количество SQL-запросов на hot-path снижено и измерено.
- [ ] Для поиска задач добавлен индексируемый механизм поиска.
- [ ] Для горячих SQL есть подтверждённые `EXPLAIN ANALYZE`.
- [ ] Для аналитики, чатов и уведомлений уменьшен размер payload и число лишних запросов.
- [ ] Есть baseline "до" и "после" с p50/p95/p99 и RPS.
- [ ] Есть регрессионные проверки, чтобы эти проблемы не вернулись.
